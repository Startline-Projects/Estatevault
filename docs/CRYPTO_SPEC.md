# EstateVault — Cryptographic Specification

**Version:** 1
**Status:** Draft v1 (locked once first vector test passes in CI on both clients)
**Owners:** Web (`/Users/xamir/Desktop/Glo Staffing/Estatevault`) — canonical. Mobile (`/Users/xamir/Desktop/Glo Staffing/estate_vault_app`) — implements to spec.
**Companion:** `docs/crypto-vectors.json` — binding test vectors. CI-blocking on both clients.

This document is the **single source of truth** for all client-side cryptography in EstateVault. Web (`@estatevault/crypto`, TypeScript + libsodium-wrappers-sumo) and mobile (`estatevault_crypto`, Dart + sodium_libs) MUST produce byte-identical outputs for every input in `crypto-vectors.json`.

Any change to this spec requires:
1. Bump `enc_version` byte.
2. Update `crypto-vectors.json`.
3. Both clients ship support for old + new versions before old is retired.
4. Migration path documented.

---

## 1. Goals

- **Confidentiality:** server (Supabase, Vercel, Next.js) never sees plaintext or any key.
- **Integrity:** ciphertext tampering detected (AEAD).
- **Cross-platform:** identical outputs in web (browser/Node) and mobile (iOS/Android).
- **Recovery:** lost passphrase recoverable via 24-word mnemonic; lose both = data gone.
- **Forward compatibility:** versioned envelopes; old data readable after upgrades.
- **Sharing:** items shareable to designated trustees without exposing MK.

## 2. Non-goals

- Defense against compromised user device.
- Defense against OS-level malware / keyloggers.
- Hiding metadata (timestamps, sizes, FK relationships, row counts). Size-padding is optional and tracked separately.
- Server-side search (replaced by client-side blind index).

## 3. Primitives

| Purpose | Algorithm | Library construct |
|---|---|---|
| Passphrase → KEK | Argon2id | `crypto_pwhash_alg_argon2id13` |
| Authenticated encryption (bytes) | XChaCha20-Poly1305-IETF | `crypto_aead_xchacha20poly1305_ietf` |
| Authenticated encryption (streams) | XChaCha20-Poly1305 secretstream | `crypto_secretstream_xchacha20poly1305` |
| Sub-key derivation | HKDF-SHA256 | RFC 5869 |
| Asymmetric (sharing) | X25519 + XChaCha20-Poly1305 | `crypto_box_curve25519xchacha20poly1305` (sealed box) |
| Signing (audit log) | Ed25519 | `crypto_sign_ed25519` |
| Blind index (search) | HMAC-SHA256 | RFC 2104 |
| Mnemonic | BIP39 (24 words, English) | standard wordlist |
| Random | OS CSPRNG | `crypto_secretbox_keygen` / `randombytes_buf` |

Any deviation = spec violation = blocked in CI.

## 4. KDF — Argon2id

Used to derive the **Key Encryption Key (KEK)** from the user passphrase.

```
KEK = Argon2id(
    password    = utf8(passphrase),
    salt        = users.kdf_salt          (16 bytes, random per user)
    memory_kib  = 65536                   (64 MiB)
    iterations  = 3
    parallelism = 2
    output_len  = 32                      (bytes)
    version     = 0x13                    (Argon2 v1.3)
)
```

### Parameter struct (stored in `clients.kdf_params` jsonb)

```json
{
  "alg": "argon2id",
  "v":   "0x13",
  "m":   65536,
  "t":   3,
  "p":   2,
  "out": 32
}
```

### Calibration policy

- Floor: `m=65536, t=3, p=2`. Refuse to derive with weaker params.
- Devices that cannot run floor in <5 s should still attempt; do not silently downgrade.
- Future bumps go through `enc_version` migration.

### Salt

- 16 bytes, generated by OS CSPRNG at signup.
- Stored in `clients.kdf_salt`.
- Never reused across users.

## 5. Master Key (MK)

```
MK = randombytes(32)        # 256-bit, generated once at signup
```

- Generated locally at signup; never transmitted unwrapped.
- Wrapped twice and stored as `clients.wrapped_mk_pass` (passphrase wrapper) and `clients.wrapped_mk_recovery` (mnemonic wrapper).
- Held in memory only after unwrap. Wiped on lock / suspend / logout.

### Wrapping

```
wrap(MK, KEK):
    nonce = randombytes(24)
    ct, tag = XChaCha20-Poly1305-IETF.encrypt(
        key   = KEK,
        nonce = nonce,
        ad    = "ev:mk-wrap:v1",
        plain = MK
    )
    return Envelope { magic="EV01", version=1, nonce, ct, tag }
```

`unwrap` = inverse with same AD; rejects on tag mismatch.

## 6. Sub-Key Derivation — HKDF-SHA256

All operational keys derive from MK using **HKDF-SHA256** with fixed `info` strings. `salt` is empty (32 zero bytes) — MK is already uniformly random.

```
deriveSubKey(MK, info) = HKDF-SHA256-Extract-and-Expand(
    salt = 0x00 * 32,
    ikm  = MK,
    info = utf8(info),
    L    = 32
)
```

### Reserved `info` strings

| Sub-key | `info` string | Use |
|---|---|---|
| DEK_db | `ev:dek:db:v1` | encrypt vault row payloads (label + data + metadata) |
| DEK_files | `ev:dek:files:v1` | encrypt file content uploaded to storage |
| INDEX_KEY | `ev:idx:hmac:v1` | HMAC key for blind-index search |
| PRIV_X25519 | `ev:key:x25519:v1` | seed for X25519 keypair (sharing) |
| PRIV_ED25519 | `ev:key:ed25519:v1` | seed for Ed25519 keypair (audit signing) |
| AUDIT_KEY | `ev:audit:hmac:v1` | HMAC for client-side audit chain |

Adding a new sub-key requires a new `info` string with `:vN` suffix; never reuse.

### Origin binding (web only, optional v1.1)

Web may extend `info` with origin: `"ev:dek:db:v1|origin=app.estatevault.com"`. If enabled, mobile spec must mirror this rule. **Default v1: no origin binding.**

## 7. Keypair Derivation

X25519 (sharing) and Ed25519 (signing) seeds are deterministic from MK so cross-platform unwrap regenerates identical keys.

```
seed_x25519  = deriveSubKey(MK, "ev:key:x25519:v1")          # 32 bytes
{pub, priv}  = X25519.keypair_from_seed(seed_x25519)         # libsodium: crypto_kx_seed_keypair OR crypto_box_seed_keypair

seed_ed25519 = deriveSubKey(MK, "ev:key:ed25519:v1")
{pub, priv}  = Ed25519.keypair_from_seed(seed_ed25519)       # libsodium: crypto_sign_seed_keypair
```

Public keys uploaded to `clients.pubkey_x25519` and `clients.pubkey_ed25519`.

## 8. Envelope Format

All ciphertext (bytes) wrapped in this fixed binary layout:

```
+--------+---------+----------+------------------+--------+
| magic  | version | nonce    | ciphertext       | tag    |
| 4 B    | 1 B     | 24 B     | N B              | 16 B   |
| "EV01" | 0x01    | random   | XChaCha20-P1305  | Poly   |
+--------+---------+----------+------------------+--------+
```

- `magic` = ASCII `"EV01"` (`0x45 0x56 0x30 0x31`).
- `version` = `0x01` for spec v1.
- `nonce` = 24 random bytes (XChaCha20 IETF).
- `tag` = 16-byte Poly1305 authenticator (appended by libsodium).
- DB column storage: `bytea` of full envelope (or split into `(ciphertext, nonce)` columns where the rest is reconstructed; see §11).

Decoders MUST reject:
- magic mismatch
- unsupported version
- length < `4 + 1 + 24 + 16`

### Associated Data (AD)

For row-level encryption, AD binds context:

```
AD = utf8("ev:vault_items:v1") || row.id (uuid bytes)
```

Prevents ciphertext from being moved between rows. Empty for the wrap-MK case which uses a fixed AD (§5).

## 9. Stream Encryption (Files)

Used for any file > 1 MiB or any video. Implementation = libsodium `secretstream`.

```
stream_init_push(key) -> (state, header[24])

for chunk in src.chunks(1_048_576):           # 1 MiB
    is_last = chunk_is_last
    out = secretstream_push(state, chunk,
                            ad     = utf8("ev:file:v1"),
                            tag    = is_last ? FINAL : MESSAGE)
    write(out)
```

Wire format on storage:

```
+------+-------+------------+-------+
| magic| ver   | header(24) | body  |
| "EVS1"|0x01  | secretstream| chunks
+------+-------+------------+-------+
```

`magic` = ASCII `"EVS1"` to distinguish from the byte envelope (`"EV01"`).

Decoders verify each chunk's tag; tampering with any chunk fails immediately.

## 10. Mnemonic Recovery

- Standard: BIP39, 24 words, English wordlist.
- Generated at signup.
- Seed → KEK_recovery:

```
seed     = bip39.mnemonicToSeed(mnemonic, passphrase="")     # 64 bytes
KEK_recovery = HKDF-SHA256-Expand(
    salt = 0x00 * 32,
    ikm  = seed,
    info = utf8("ev:kek:recovery:v1"),
    L    = 32
)
wrapped_mk_recovery = wrap(MK, KEK_recovery)
```

Recovery flow: user enters 24 words → derive KEK_recovery → unwrap MK → set new passphrase → re-wrap as `wrapped_mk_pass`.

**Lost passphrase + lost mnemonic = MK is unrecoverable. Document loudly in UX.**

## 11. Database Storage Format

| Column | Type | Contents |
|---|---|---|
| `clients.kdf_salt` | `bytea` | 16 raw bytes |
| `clients.kdf_params` | `jsonb` | param struct §4 |
| `clients.wrapped_mk_pass` | `bytea` | full envelope §8 |
| `clients.wrapped_mk_recovery` | `bytea` | full envelope §8 |
| `clients.pubkey_x25519` | `bytea` | 32 bytes |
| `clients.pubkey_ed25519` | `bytea` | 32 bytes |
| `clients.enc_version` | `smallint` | spec version (1) |
| `vault_items.ciphertext` | `bytea` | ciphertext + tag (no magic/version/nonce) |
| `vault_items.nonce` | `bytea` | 24 raw bytes |
| `vault_items.enc_version` | `smallint` | spec version (1) |
| `vault_items.label_blind` | `bytea` | HMAC-SHA256(INDEX_KEY, normalize(label)) |

For row-level fields the magic + version are implicit (column type and `enc_version` carry that). Full envelope assembly on decode:

```
envelope = "EV01" || enc_version || nonce || ciphertext_with_tag
```

Storage objects (files): full envelope (§8) or stream format (§9) written to bucket. Path pattern: `vault/{client_id}/{uuid}.bin` (no extension leak).

## 12. Sharing Protocol (Trustees)

Per-item DEK + sealed box to recipient's pubkey.

```
# Owner side (creating share):
item_dek           = deriveSubKey(MK, "ev:dek:db:v1")        # OR per-item random key (recommended hardening)
recipient_pub      = fetch from /api/crypto/pubkey
wrapped_dek_for_recipient = crypto_box_seal(item_dek, recipient_pub)
                                                              # ephemeral sender keypair, recipient-only decrypt
                                                              # forward-secret per share

POST /api/share { item_id, recipient_user_id, wrapped_dek, sender_pubkey=null (sealed box) }
```

```
# Recipient side (reading share):
GET /api/share/incoming
for s in shares:
    dek = crypto_box_seal_open(s.wrapped_dek, recipient_pub, recipient_priv)
    plaintext = decrypt vault row using dek
```

**Recommended hardening (v1.1):** generate a random per-item DEK at creation, encrypt the item with it, and wrap that DEK for the owner with `MK`. Sharing then wraps the same per-item DEK to recipients. Compromise of one item's DEK does not affect others.

## 13. Blind Index (Search)

Searchable fields (e.g. vault_items.label) get a deterministic HMAC stored alongside ciphertext:

```
normalize(s) = NFC-normalize, lowercase, trim whitespace, collapse internal whitespace to single space
label_blind  = HMAC-SHA256(INDEX_KEY, utf8(normalize(label)))[:16]   # 16-byte truncation
```

Search:
```
client computes blind = HMAC-SHA256(INDEX_KEY, utf8(normalize(query)))[:16]
POST /api/vault/search { label_blind: blind }
server returns rows matching exact blind value
```

Tradeoffs:
- Equality only (no substring/prefix). Substring needs n-grams (out of scope v1).
- Reveals frequency to anyone with DB read access (mitigate with per-field salt in v1.1).

## 14. Audit Log

Each mutating client action signs a deterministic record:

```
record = {
  ts:        ISO-8601,
  actor:     auth.uid(),
  action:    "vault.item_create",
  resource:  vault_item.id,
  hash_prev: prev_record_hash,
  ciphertext_hash: SHA-256(envelope)
}
sig = Ed25519.sign(PRIV_ED25519, canonical_json(record))
POST /api/audit { record, sig }
```

Server stores `record + sig`. Tamper-evident chain via `hash_prev`. Verifiable client-side using user's `pubkey_ed25519`.

## 15. Key Lifecycle / Rotation

| Event | Action |
|---|---|
| Passphrase change | re-derive KEK with new passphrase + same salt; re-wrap MK; update `wrapped_mk_pass` |
| Mnemonic regenerate | new mnemonic → new KEK_recovery → re-wrap; update `wrapped_mk_recovery` |
| MK rotation (rare; on suspected compromise) | new MK → re-encrypt all rows + storage; expensive; requires backfill-style job |
| Sub-key version bump | introduce `:v2` info; backfill or dual-read until cutover |
| Spec version bump | new `enc_version`; clients support N and N-1 simultaneously through migration |

Routine 90-day re-wrap of MK with same passphrase is OPTIONAL and adds limited value; skip for v1.

## 16. Threat Model Summary

| Adversary | Defense |
|---|---|
| Supabase admin / DB dump | Sees ciphertext only; no key material derivable |
| Vercel / Next.js host | API routes never touch plaintext or MK; ciphertext in transit |
| Network attacker | TLS + ciphertext payloads |
| Other users | RLS + per-row AD binding + sharing only via wrapped DEK |
| Lost passphrase | Mnemonic recovery |
| Lost passphrase + mnemonic | Data lost (acceptable per design) |
| Compromised user device | Out of scope |
| Server compelled to backdoor | Open-source crypto packages enable verification builds |

## 17. Test Vector Format (`crypto-vectors.json`)

Both implementations consume this file and assert byte-identical outputs. CI pipelines on both repos must run vector tests on every PR.

See `docs/crypto-vectors.json`. Vector categories:

| Category | What it pins |
|---|---|
| `kdf` | Argon2id outputs for fixed (passphrase, salt, params) |
| `hkdf` | HKDF-SHA256 outputs for each `info` string |
| `aead_bytes` | XChaCha20-Poly1305-IETF encrypt/decrypt for fixed (key, nonce, ad, plaintext) |
| `aead_stream` | secretstream for chunked input |
| `envelope` | Full encode/decode of byte envelope |
| `mnemonic` | BIP39 seed + KEK_recovery for fixed mnemonic |
| `keypair` | X25519 + Ed25519 derivation from seed |
| `sealed_box` | Sender-anonymous wrap to recipient |
| `blind_index` | HMAC-SHA256 truncated to 16 bytes |
| `audit_sig` | Ed25519 sign + verify for canonical record |

Vector file is **frozen** at v1 release; any change = new file (`crypto-vectors-v2.json`) and `enc_version=2`.
