Critical✅ FIXEDBUG-1 Webhook miss = paid but never fulfilled▶
In plain terms
Payment is confirmed by a single message from Stripe (the "webhook"). If that message is missed, delayed, or errors, the card is charged but the order never moves forward and documents never generate. Customer paid, gets nothing — the highest-trust failure.
Where
Stripe webhook handler
How to check
In staging, disable the webhook endpoint, then complete a test payment → the order stays stuck pending/paid and no documents appear.
Fix
Make the handler idempotent and retried; add a reconciliation job that polls Stripe for completed sessions whose orders are still pending; alert on paid-but-unfulfilled orders past a threshold. (Closely related to new BUG-13.)
Resolution (2026-06-09)
Webhook is now idempotent — claimEvent(event.id) dedups true duplicates while re-running crashed/partial attempts, then marks completed/failed. Reconciliation cron app/api/cron/reconcile-orders/route.ts checks the end result (are the PDFs there?) independent of generation path: re-dispatches the replay-safe handler for paid orders that never ran, triggers generation for orders stuck in generating, and alerts admin on anything unfinished past 60 min. Admin retry-fulfillment route for manual recovery. (Commit 1b7274f.)

Critical✅ FIXEDBUG-8 Webhook marks event "done" before doing the work → failed orders never retry — ✅ FIXED▶
In plain terms
The payment webhook records "I handled this event" before it actually fulfills the order. If fulfillment then fails halfway, Stripe's automatic retry sees the "already handled" marker and skips it — so the order is permanently stuck. This is worse than BUG-1: the safety net (retry) is disabled by the bug itself.
Where
app/api/webhooks/stripe/route.ts:42-49; stripeWebhookRepo.ts:8-14
Impact
Customer charged, fulfillment frozen forever, auto-recovery silently consumed. Underlies BUG-23/65/66/68/69.
Check on website
In staging, break a write inside the doc-checkout handler, pay a checkout → charged but no documents; resending the Stripe event does nothing (returns "duplicate").
Fix
Mark the event processed only AFTER the handler succeeds (or a received→processed two-phase state, or one transaction); on failure return non-2xx so Stripe redelivers. Add a reconciliation cron.
Resolution
Two-phase guard in route.ts + stripeWebhookRepo.ts: claimEvent claims the event as processing (only a prior completed row is a true duplicate); markCompleted runs only after the handler returns, markFailed + HTTP 500 on throw so Stripe redelivers into a clean re-run. stripe_webhook_events carries status/completed_at/last_error; event_id PK backs the claim. Reconcile cron (app/api/cron/reconcile-orders/route.ts, every 15 min) re-runs paid-but-unfulfilled orders. Verified live: 265 completed / 2 failed (retryable) / 0 poisoned. Handler re-runs are now safe per BUG-23 (per-side-effect idempotency).
High✅ FIXEDBUG-9 Stripe failure leaves an orphaned "pending" order — ✅ FIXED▶
In plain terms
At checkout the system saves the order in the database first, then asks Stripe to set up payment. If Stripe fails (bad key, outage, network blip), the request errors out — but the half-finished "pending" order is already saved with no payment attached. These junk orders pile up and pollute your reports; retries make duplicates.
Where
lib/checkout/createCheckoutSession.ts (~line 226)
How to check
In staging, set an invalid STRIPE_SECRET_KEY and run a will/trust checkout → the request fails, but a pending order with no stripe_session_id appears in the orders table.
Fix
Create the Stripe session first, then save the order — or wrap it and delete/mark-failed the order if Stripe errors. Add a job to expire stale pending orders.
Resolution
The Stripe sessions.create() call in createCheckoutSession.ts is wrapped in try/catch — on failure it deletes the just-created order (orderRepo.deleteById) and returns HTTP 502 with a friendly retry message, leaving no orphan. Safe because the order has nothing attached yet (payment/docs/payouts happen later). A cleanup sweep in app/api/cron/reconcile-orders/route.ts deletes pending orders with null stripe_session_id older than 30 min (logs ids first — no silent mass delete). Verified live on staging: the sweep matches a true orphan but NOT an abandoned-with-session order nor a fresh in-flight one.
High✅ FIXEDBUG-10 Document generation failure leaves the order stuck forever — ✅ FIXED▶
In plain terms
After payment the order is marked "generating" and the AI builds the documents. If that AI step fails (API error, timeout, rate limit), the documents stay stuck and nothing retries or flags it. Customer paid, no usable output, no one notices.
Where
Doc-gen pipeline (Claude API) + documents table
How to check
In staging, force the Claude API call to fail during generation → the documents rows stay pending/generating indefinitely with no retry.
Fix
Retry with backoff; add an admin alert/dashboard for orders stuck in generating; allow a manual re-trigger. (Closely related to the new BUG-13.)
Resolution
Failure is now honest and recoverable. (1) On a generation error the worker marks that document failed and does not advance the order to delivered — it stays generating so a safety net can catch it (app/api/documents/process/route.ts:202-212 and the queue path :331-353). (2) Auto-retry: app/api/cron/reconcile-orders/route.ts runs every 15 min (vercel.json cron */15 * * * *), finds orders stuck in generating/failed/pending with missing PDFs and re-fires process-now, which re-runs all doc types so previously failed docs regenerate. (3) Admin alert: sendFulfillmentFailureAlert for anything unfinished past 60 min. (4) Admin dashboard: app/api/admin/orders-missing-docs/route.ts lists every stuck order with a failureKind tag. These areas were audited and found correct — here's why each is safe, so you know it was actually checked.

  • Prices & revenue splits are correct everywhere. Every charge is calculated on the server from one source-of-truth file (lib/orders/pricing.ts); the browser can't change the amount. Standard/enterprise splits match the rules and always add up to the full price.
  • Partners can't be double-paid. Each payout uses a unique key (e.g. transfer_partner_<orderId>), so even if Stripe retries, the partner is paid only once.
  • The Stripe webhook ignores duplicate events. Repeated/retried notifications for the same event are detected and skipped, so fulfillment doesn't run twice.
  • Clients, partners and admins are kept apart. The database enforces row-level security (each user only sees their own rows) and the app redirects each role to its own portal — a client can't reach /pro or /sales.
  • Document downloads check ownership. A download is only allowed if the document belongs to the logged-in client, and the attorney-review gate blocks download until review is delivered.
  • Vault items are locked to each owner. Every vault read/edit is scoped to the logged-in client; a stale session with no profile is signed out automatically (BUG-26 handling).
  • The "complexity" flag is display-only — by design, not a bug. It only shows the customer a suggestion; it doesn't gate anything, so it doesn't need server enforcement. (Per your confirmation.)
