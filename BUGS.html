CriticalBUG-8 Acknowledgment signature is faked server-side▶
In plain terms
Before anyone gets documents they're supposed to check an "I understand this isn't legal advice" box. The system always records that they signed it — even when the box was never shown. Your records claim a signature that never happened.
Where
lib/checkout/createCheckoutSession.ts:188-189 (also :351-352, :416-420); app/will/checkout/page.tsx
Impact
Documents generate for customers who never acknowledged; the order record is fabricated proof. Compliance + legal exposure (Core Rule 3).
Check on website
Start a will, go to checkout with a real card — you're never shown an "agree" step on the pay path. Then look at the order in the dashboard / Supabase orders table: acknowledgment_signed = true anyway.
Fix
Require a real acknowledged: boolean in the checkout schema; reject when false; set the column from that value, not a hardcoded true. Show the acknowledgment on the paid path.
CriticalBUG-9 Hard stops (safety brake) not enforced on the purchase paths▶
In plain terms
Some cases (special-needs child, irrevocable trust) must stop and go to a real attorney — "no override." That stop only lives in the quiz page as browser state. Customers can start a will or trust straight from the homepage/footer, where the risky questions are never asked, and nothing on the server re-checks. So a family that legally needs an attorney can pay and get do-it-yourself documents.
Where
app/quiz/page.tsx:118-128 (only place it exists); app/will/page.tsx, app/trust/page.tsx; no check in createCheckoutSession.ts or handleDocumentCheckout.ts:274-322
Impact
The advertised "cannot be overridden" safeguard doesn't exist on the paid paths. The webhook is the last possible checkpoint and does no stop check (Core Rule 4).
Check on website
From the homepage click Trust (not the quiz) → go through every intake screen → it never asks about a special-needs dependent or irrevocable trust → reach checkout → it lets you pay. The "see an attorney" screen never appears.
Fix
Ask the hard-stop questions inside /will and /trust, AND re-derive them from the intake answers server-side in checkout and again in the webhook before generating — block with an attorney referral. Never trust a client flag.

CriticalBUG-17 Platform pays out more attorney-review fee than it collected▶
In plain terms
The customer always pays a fixed $300 for attorney review. But an attorney partner can set their own "review fee" to $1,000+, and the system pays the partner that full amount — so EstateVault pays out $1,000 while only collecting $300. Direct money loss of $700+ per review. A safety range exists in the code but is never actually used.
Where
lib/attorney-review/routing.ts:88; lib/webhooks/stripe/handleAttorneyReview.ts:57-69; lib/checkout/createCheckoutSession.ts:148; ATTORNEY_REVIEW_FEE_RANGE in pricing.ts:27 is referenced nowhere.
Impact
Net loss out of platform funds on every attorney-reviewed order where the partner set a high custom fee.
Check on website
As an attorney partner go to Pro → Settings, set the review fee to $1,000, save. Have a test client buy a Will with Attorney Review (pays $300). In Stripe → Connect → that partner → Transfers, the fee transfer is $1,000 vs the $300 collected.
Fix
Clamp the custom fee to the defined range at every save point; never transfer more than was collected — if a partner's fee exceeds $300, charge that at client checkout instead of the fixed price.

KnownBUG-1 Stripe failure leaves an orphaned "pending" order▶
In plain terms
At checkout the system saves the order in the database first, then asks Stripe to set up payment. If Stripe fails (bad key, outage, network blip), the request errors out — but the half-finished "pending" order is already saved with no payment attached. These junk orders pile up and pollute your reports; retries make duplicates.
Where
lib/checkout/createCheckoutSession.ts (~line 226)
How to check
In staging, set an invalid STRIPE_SECRET_KEY and run a will/trust checkout → the request fails, but a pending order with no stripe_session_id appears in the orders table.
Fix
Create the Stripe session first, then save the order — or wrap it and delete/mark-failed the order if Stripe errors. Add a job to expire stale pending orders.

Checked & sound (no action needed)

These areas were audited and found correct — here's why each is safe, so you know it was actually checked.

  • Prices & revenue splits are correct everywhere. Every charge is calculated on the server from one source-of-truth file (lib/orders/pricing.ts); the browser can't change the amount. Standard/enterprise splits match the rules and always add up to the full price.
  • Partners can't be double-paid. Each payout uses a unique key (e.g. transfer_partner_<orderId>), so even if Stripe retries, the partner is paid only once.
  • The Stripe webhook ignores duplicate events. Repeated/retried notifications for the same event are detected and skipped, so fulfillment doesn't run twice.
  • Clients, partners and admins are kept apart. The database enforces row-level security (each user only sees their own rows) and the app redirects each role to its own portal — a client can't reach /pro or /sales.
  • Document downloads check ownership. A download is only allowed if the document belongs to the logged-in client, and the attorney-review gate blocks download until review is delivered.
  • Vault items are locked to each owner. Every vault read/edit is scoped to the logged-in client; a stale session with no profile is signed out automatically (BUG-5 handling).
  • The "complexity" flag is display-only — by design, not a bug. It only shows the customer a suggestion; it doesn't gate anything, so it doesn't need server enforcement. (Per your confirmation.)
