1 · B7 — Give the app's URL a single name

B7 ~12 files Quick · low risk

What it is. The app's own web address is read from an environment variable, but the code uses three different names: NEXT_PUBLIC_SITE_URL, NEXT_PUBLIC_APP_URL, NEXT_PUBLIC_BASE_URL.

Why it matters. Not just messy — a real bug. A password-reset or verify link can be built from the wrong domain depending on which file runs, so a user could get a link that points nowhere.

How to do it
  1. Make one tiny helper lib/config/appUrl.ts:
export function getAppUrl(): string {
  return (
    process.env.NEXT_PUBLIC_SITE_URL ||
    process.env.NEXT_PUBLIC_APP_URL ||
    process.env.NEXT_PUBLIC_BASE_URL ||
    "https://www.estatevault.us"
  );
}
  • Find every use: grep -rn "NEXT_PUBLIC_SITE_URL\|NEXT_PUBLIC_APP_URL\|NEXT_PUBLIC_BASE_URL" app lib (~12 files).
  • Replace each with getAppUrl().
  • In .env.example, keep only NEXT_PUBLIC_SITE_URL; delete the other two.

2 · B4 — Finish the shared login/permission check

B4 7 → 35 routes Quick · low risk

What it is. Routes that need a logged-in user used to each write their own "is this person allowed?" code. The shared helper requireAuth() now does this — 35 routes use it — but a handful still hand-roll the check.

Why it matters. When the rule changes (add a role, an audit log, a lockout), you want to edit one place, not chase leftovers. Hand-rolled checks also return inconsistent error messages that confuse the frontend.

How to do it
  • List the stragglers: grep -rln "auth.getUser()" app/api, minus the ones already on requireAuth.
  • Replace the manual getUser() + profiles.user_type lookup with const auth = await requireAuth(["partner","admin"], req); if ("error" in auth) return auth.error;
  • Use auth.admin / auth.user / auth.profile from the helper instead of re-creating them.
  • Make sure the route is wrapped in withRoute(...) like its neighbors.

3 · B3 — Send all email through one place

B3 28 → 1 init Step 1 done · branding left

✅ Step 1 done (2026-06-02): all 15 inline new Resend(...) replaced with the shared getResend() from lib/email.ts (28 → 1 client init). Verified green. Remaining: the from-address + branded-template step below — lower value, needs live email testing (it changes retry/error behavior).

What it was. 15 routes created their own email client (new Resend(...)) and sent directly, instead of going through the shared lib/email.ts.

Why it matters. Change the sender domain or branding and today you edit ~15 files. lib/email.ts already has branded templates — the routes just bypass it.

How to do it
  • Find them: grep -rln "new Resend(" app/api.
  • For each email type, add/reuse a function in lib/email.ts (e.g. sendPartnerWelcomeEmail({...})).
  • Replace the inline resend.emails.send(...) with a call to that function; remove the inline new Resend(...) and hardcoded from-address.
  • Send a test of each email type (Resend test mode) before shipping.

4 · B5 — One shared "call the API" helper

B5 ~50 screens on api-client ✅ Done (verified)

✅ Done (verified 2026-06-02): already resolved by the Jun-2 refactor. The typed lib/api-client/ layer (11 modules over authedFetch) is imported by ~50 screens; 0 raw internal /api/ fetches remain. The audit's "2 files use authedFetch" counted only direct calls — the wrappers route through it, so token-refresh reaches every screen. No code change needed; detail kept for reference.

What it was. Screens made ~122 raw fetch("/api/...") calls. The wrapper that refreshes an expired login and retries (lib/api/authedFetch.ts) was used in just 2 files. Typed wrappers live in lib/api-client/.

Why it matters. With raw fetch, a user whose token quietly expired gets a random logout or silent failure instead of a smooth refresh-and-retry. No shared loading/error handling either, so every screen reinvents it.

How to do it
  • Standardise on lib/api-client/* typed wrappers (preferred — they add types too) built on authedFetch.
  • Per feature area, add small client functions, e.g. lib/api-client/partners.ts → listPartners(), createPartner(...).
  • Replace the screen's raw fetch("/api/partners") with listPartners().
  • Do this alongside item 5 (B2) — when a screen gets an API route, give it a client function at the same time.

5 · B2 — Stop screens talking to the database directly ⭐

B2 done Big · ✅ complete · phased

✅ Done (2026-06-03, commit 9a0d97a): All four phases complete — every browser-side screen goes repo → API route → lib/api-client/* → screen, never supabase directly. Converted ~30+ screens: pro/onboarding (steps 1-7 + 3 vault variants, incl. logo uploads + promo comp), pro/settings, pro/revenue, pro/{support,preview,training,clients,vault-clients,documents,dashboard}, sales + pro/sales partner-detail twins, sales/commission, sales/partners, sales/dashboard, pipelines, client dashboard (funding-checklist, settings), dashboard/documents, attorney reviews/pipeline, and the sales/pro-sales/attorney portal layouts (getMyProfile()). ~25 new endpoints. Security hardened: promo fee-waiver server-validated on both partner-self and sales-rep paths; financial fields excluded from the self-update whitelist; logo upload server-side; sales partner-detail ownership via created_by (foreign id → 404). e2e anon guards for every new endpoint. Verified: tsc clean (1 pre-existing quizSessionRepo error untouched), 376 unit, lint clean. Left by design: app/pro/layout.tsx + app/dashboard/page.tsx — server components on the RLS-scoped server client; admin repos would bypass RLS for no browser-leak benefit. Open flag: custom_review_fee is partner-editable in the self-update whitelist — lock server-side if the fixed-pricing rule should cover it.

What it is. 57 UI screens run database queries (supabase.from(...).select/insert/update) inside the screen itself — no API route in between. The only thing protecting the data is row-level security (RLS).

Why it matters.

  • Business rules live inside screens — no single place to add logging, caching, validation, rate-limiting.
  • A second frontend (mobile app) couldn't reuse any of it — full rewrite.
  • One weak RLS policy = direct data exposure with no server-side backstop.
How to do it — phase it, NOT all 53 at once

For each screen, the same three layers (copy how the vault feature already does it):

  1. Repo — put the query in lib/repos/server/<thing>Repo.ts.
  2. API route — add app/api/<thing>/route.ts: requireAuth → validate (Zod) → call repo → ok(...).
  3. Client + screen — add a lib/api-client/ function (this is B5), then change the screen to call it instead of touching supabase.
Suggested phases (one area per PR)
  • Phase A — app/pro/* (biggest cluster: dashboard, clients, revenue, 11 onboarding steps). Highest payoff.
  • Phase B — app/sales/* + app/pro/sales/* (pipeline, partners, commission).
  • Phase C — dashboard / attorney / checkout-success screens.
  • Phase D — the components/* that query the DB (DocumentActions, PackageStatusCard, …).
Per-phase checklist
  • Move each query into a repo function.
  • Wrap it in an API route with requireAuth + Zod.
  • Add a client function; switch the screen to it.
  • Screen no longer imports the browser supabase client for data.
  • tsc + lint + test green; click through the screen in the browser.

Suggested order & rough timeline

  1. Week 1 (quick wins): B7 (app URL) → finish B4 (login check) → start B3 (email).
  2. Week 2: finish B3 → set up the lib/api-client/ standard for B5.
  3. Weeks 3+: B2 phase by phase (A → B → C → D), bringing B5 client functions along with each phase.
Quick wins (B7, B4, B3) are safe and give a fast, visible structural win. B2 + B5 is the real project — one area at a time, verifying each in the browser before moving on. None of these are live bugs today; they're scale & security hardening.
Never under "cleanup": don't change the fixed prices, the revenue-split math, or the hard-stop rules (special-needs dependent, irrevocable trust). Those are business law in CLAUDE.md.
