Start here — the whole thing in plain English

🏠 Think of the app like a big office building

Right now, every room built its own front door, its own lock, its own security guard, and its own filing cabinet. 119 rooms (the API routes), each re-doing the same setup slightly differently. When you want to change one rule — say "check ID more carefully" — you have to walk into all 119 rooms and fix each guard by hand. Miss one, and that room is now a security hole.

This plan builds one shared lobby: one front desk that checks ID (auth), one rulebook (validation), one records department that touches the files (the data layer), and one front-desk phone line the app uses to call the building (the API client). Every room just connects to the lobby. Change a rule once, and all 119 rooms get it. That's what "scalable" means here — add the 120th room without rebuilding the plumbing.

Why this matters to you Today the code works, but it's fragile and slow to change because the same things are copy-pasted everywhere. After this refactor, the app does the same things for your users — but it's faster to add features, harder to introduce bugs, and a fix in one place actually fixes everything.

Word list (so the rest makes sense)

API route
A single backend endpoint the app calls, e.g. "save a vault item." There are 119 of them. Each lives in a route.ts file.
Auth / requireAuth
Checking who is making the request and whether they're allowed. The "security guard at the door."
Admin client
A special database connection that can bypass normal restrictions. Powerful — should be created in one place, not copied into 64 files.
Repo (repository)
The one piece of code allowed to read/write a database table. The "records department." Today routes skip it and grab files themselves.
Validation / Zod
Checking that incoming data has the right shape before trusting it ("is this actually an email?"). The "rulebook at the desk."
.from("table")
A raw database query written directly inside a route. There are 581 of these scattered around — the thing we want to centralize.
fetch
How the screen (browser) calls the backend. Today copy-pasted in many components, each handling errors its own way.
RLS
Row-Level Security — database rules that stop user A from seeing user B's data. Easy to bypass by accident with the admin client.
DEK / crypto
The encryption keys that lock each user's private vault. The most sensitive code — a mistake here means lost or exposed data.
Migration
A versioned change to the database structure. They should apply in a clear order; right now two competing systems exist.
The "gate"
Three automatic checks — tsc (type errors), lint (style/rules), test — that must pass before moving on. The safety net.
SSOT
Single Source of Truth — one place a fact (like a price) is defined, so it can't disagree with itself across the app.

Target architecture

A thin, repeatable request pipeline. Each layer is written once and reused; new features compose layers instead of re-implementing them.

  Browser / Mobile  ─►  Typed API client  (lib/api-client/*)        ← one fetch wrapper, typed
                        authedFetch · typed endpoints · envelope     responses, 401 refresh, 1 error shape
                                     │
  app/api/**/route.ts ─►  Route handler (THIN)                       ← no inline admin client,
                        withRoute(): auth → validate → service         no inline query, no inline zod
                                     │
                        Service / domain layer  (lib/<domain>)         ← business rules: pricing,
                        vault · checkout · trustee · partner …         splits, hard stops
                                     │
                        Server repos  (lib/repos/server/*)             ← the ONLY place that
                        all .from("table") lives here                  touches Supabase tables
                                     │
                        Supabase (Postgres + RLS)

  Shared kernel (every layer): requireAuth · createAdminClient · zod registry ·
  lib/crypto/encoding.ts (one b64/bytea) · lib/orders/pricing.ts (SSOT) · types/db.generated.ts
Reading the diagram top to bottom A request flows downward through layers, each with one job: the screen calls through one phone line → the route just greets and directs (no real work) → the service applies business rules → the repo is the only thing that touches the database → the database stores it. The bottom strip ("shared kernel") is the stuff every layer borrows instead of re-making: the ID checker, the rulebook, the price list, the data types.

Design principles

One way to do each thing. A route never re-declares auth, an admin client, validation, or a query — it calls the shared layer.
Thin routes, fat services, single data layer. Tables touched only in lib/repos; rules in lib/<domain>; routes orchestrate.
Validate at the boundary, type through the stack. Every body/query parsed by a named Zod schema; generated DB types make a column rename break the build, not production.
One client transport. Components call a typed API client, never raw fetch. One place owns auth refresh + error envelope.
Single source of truth for money & config. Prices, promo codes, splits live in one module. Never inlined.
Make wrong code not compile / not lint. ESLint-error rules + return types + CI so drift can't re-accumulate.
Never touch pricing/split/hard-stop values under "cleanup." Relocate, don't change (CLAUDE.md law).

Phased plan

Each phase is independently shippable and gated by npx tsc --noEmit && npm run lint && npm test. Foundations land before the mass migrations that depend on them. Big sweeps run route-group by route-group, not all at once.

Phase 0 Safety net do first

Pin behavior on the seams later phases will move.

  • Generate Supabase types → types/db.generated.ts (wire npm run db:types). Foundation for every typed layer.
  • Add characterization tests on the flows Phase 2 refactors: vault items CRUD, checkout → webhook, trustee unlock.
Payoff: later phases can prove they preserved behavior.
In simple words Before changing anything, take photos of how things work now (tests) and make the database describe its own shape to the code (types). So later, if a change accidentally breaks something, an alarm goes off immediately.

Phase 1 Establish the shared kernel

Build the single implementations everything else calls. Low-risk, no mass migration yet.

  • lib/api/route.ts — a withRoute() wrapper: try/catch + generic error, enforced Promise<NextResponse>, auth + Zod hooks. The standard handler shape.
  • Confirm requireAuth = only auth path, createAdminClient = only admin client. Make UserType the SSOT (add the real DB role strings).
  • lib/api/response.ts — ok(data) / fail(msg, status) envelope.
  • lib/crypto/encoding.ts — consolidate the 7 b64/bytea copies; migrate call sites.
  • lib/orders/pricing.ts — PRICES + PROMO_CODES registry. Move inlined values here. Change no value.
Payoff: the toolkit exists — from here, migrating a route removes code.
In simple words Build the shared lobby — one ID checker, one rulebook, one error format, one price list. Nothing else changes yet; we just create the shared tools so every room can start using them next.

Phase 2 Server data-access layer biggest win

581 .from() calls live inside routes today. Move them behind server repos so tables are touched in one layer.

  • Create lib/repos/server/ (current repos are client-only / import browser code — a server split is required).
  • Per group (vault → checkout → documents → partner → sales → trustee → admin): extract inline queries into typed repo functions. Route becomes withRoute → requireAuth → parse → repo.x() → ok().
  • Bake ownership scoping into repo writes (.eq("client_id", …)) so authorization is structural.
Payoff: a schema/RLS/ownership change = one repo edit, not 50 routes; routes shrink ~60–70%.
In simple words The 581 places that reach into the database directly get moved into one tidy "records department." Done one section at a time (vault first). This is the biggest change — and the biggest payoff — so it gets the most testing.

Phase 3 Validation at every boundary

  • Grow lib/validation/schemas.ts into a per-domain registry: vaultItemSchema, trusteeSchema, farewellSchema, checkoutSchema, partnerClientSchema, amendment…
  • Wire withRoute to safeParse body/query, reject early with the standard envelope. Reconcile the duplicate WillIntake interface vs willIntakeSchema.
Payoff: no route trusts client shape; one validation style.
In simple words Every door checks incoming data against the rulebook before trusting it — so bad or malicious input is rejected at the entrance, the same way everywhere.

Phase 4 One typed client transport kills fetch-everywhere

  • lib/api-client/ — typed client on the existing authedFetch: functions per endpoint returning typed data, throwing one ApiError.
  • Migrate components/pages off raw fetch("/api/…"), group by group. Shared async-state hook for loading/error/success.
Payoff: components drop ~20–30% boilerplate; auth refresh + error shape owned once; renaming a route updates one client fn.
In simple words The screens stop each writing their own way to call the backend. One shared phone line handles every call, deals with expired logins, and reports errors the same way — so pages get smaller and behave consistently.

Phase 5 Single-source config cleanup

  • Replace every remaining inlined price/promo with imports from lib/orders/pricing.ts.
  • Tighten calculateSplit(productType: ProductType) to a union — no silent all-zeros on a typo.
Payoff: price/promo change = one edit.
In simple words Prices and promo codes live in one file instead of being typed into ~15 places. Change $400 once, everywhere updates. (The numbers themselves are not changed — just moved.)

Phase 6 Type system hardening

  • Flow types/db.generated.ts through repos/services; remove ad-hoc per-route row types.
  • Add explicit Promise<NextResponse> to all handlers (trivial via withRoute). Remove stray as any CSS casts.
Payoff: a column/enum change fails the build.
In simple words Make the code and the database agree on exact data shapes. If someone renames a database column, the code refuses to build instead of silently breaking in front of a user.

Phase 7 Migration lineage baseline

  • Squash 18 legacy migration-*.sql + database.sql into one timestamped baseline; keep only the 28 dated migrations after it.
Payoff: reproducible rebuilds; no lexical-order landmine.
In simple words The database's change-history is currently a messy mix of two systems. Tidy it into one clean ordered list, so rebuilding the database from scratch always works the same way.

Phase 8 Lock it in (prevent re-drift)

  • Promote ESLint rules warn → error: ban raw fetch("/api/…") in components, ban inline createAdminClient, ban .from() outside repos, require requireAuth in routes.
  • Add a CI step running the gate on every PR. Update stale docs (new-api-route.md, app/CLAUDE.md, conventions.md).
Payoff: the architecture becomes the path of least resistance; drift can't silently return.
In simple words Turn the new rules into automatic enforcers. If a future developer tries to go back to the old copy-paste habits, the tools block it — so the cleanup stays clean.

Will this break functionality?

Honest answer: no refactor of this size is zero-risk. The plan is built to keep risk low and catchable before shipping — but the safety lives in the discipline, not the document.

Mostly behavior-preserving — with deliberate exceptions

The intent is to move code, not change what it does. A few steps intentionally change behavior because the current behavior is buggy:

  • Phase 1: UserType gains the real DB role strings → role gating starts matching where it silently failed.
  • Phase 2: ownership scoping baked into repo writes → loosely-scoped writes get tighter.

Fixes, not regressions. The honest claim is "no unintended loss of working features," not "no change at all."

Risk is not uniform

PhaseRiskWhy
0 (types+tests), 5 (pricing move), 8 (lint/CI)Lowadditive or mechanical; no value changes
1 (kernel), 3 (zod), 6 (types)Low–Mednew code; old code keeps working until switched over
7 (migration squash)Medrebuild correctness — test on a fresh DB
2 (server repos, 581 queries), 4 (fetch sweep)Highestmost surface touched; subtle bugs hide here

What keeps it safe

  • Characterization tests first (Phase 0) — prove a flow behaves the same before/after.
  • Per-group migration, never big-bang — a mistake breaks one group, not the app.
  • Verify gate green before each next step.
  • No pricing / split / hard-stop value edits.

Gaps you must respect (where it CAN bite)

  • The gate won't catch everything. Stripe webhooks, RLS, auth cookies, crypto/DEK, trustee access need live secrets → verify with e2e + manual run.
  • Tests only protect what they cover. Money/crypto/auth flows need real tests before Phase 2 touches them.
  • Crypto/DEK + money = danger zone. A slip there is silent data loss / mis-pay, not a loud crash. Do those groups last, with most coverage.
Bottom line: done as written — small steps, tests first, per-group, gate each step, verify money/crypto/auth flows live — the chance of breaking working functionality is low and catchable before shipping. Skip those steps and it becomes high.

Guardrails (every phase)

  • Never alter pricing, revenue-split, or hard-stop values under cleanup — relocate, don't edit (CLAUDE.md).
  • No public-shape change without updating every caller in the same step. Half-migrated is worse than not migrated — finish a route group before moving on.
  • No any / @ts-ignore / as to make the gate pass.
  • Small reviewable commits per group; gate green before the next.
