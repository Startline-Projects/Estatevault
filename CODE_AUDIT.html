Critical

C-1 Critical Stripe webhook has no idempotency; retries double-pay partners & affiliates ✅ Fixed

app/api/webhooks/stripe/route.ts:44-51 · lib/repos/server/stripeWebhookRepo.ts · lib/stripe-payouts.ts

The handler never records event.id or checks if an event was already processed. Stripe retries on any non-2xx/timeout, and checkout.session.completed runs many sequential awaits. A re-delivery re-runs transferToPartner / transferToAffiliate and re-inserts payouts / orders rows. No idempotencyKey on any stripe.transfers.create.

Blast radius: double-paying partners and affiliates real money; duplicate payout/order rows; duplicate generation jobs.
Fix: add a processed_stripe_events(event_id pk) table; insert event.id at the top of the handler, return 200 if it exists. Pass { idempotencyKey: `transfer_${orderId}` } to every transfers.create.
✅ Implemented 2026-06-02: every stripe.transfers.create now passes an idempotencyKey — transfer_partner_${orderId}, transfer_affiliate_${orderId}, and a caller-supplied/derived key for the affiliate batch (lib/stripe-payouts.ts). Stripe itself now rejects a duplicate transfer even if the handler crashes mid-way and retries. Covered by tests/unit/stripe-transfer-idempotency.test.ts.

C-2 Critical DEK wrapped with no AAD → cross-tenant vault decryption via one DB column copy ✓ verified ✅ Fixed

lib/crypto/keyManager.ts:25-31 (wrapKey/unwrapKey, no AAD) · lib/api/dek.ts:51,55,80

Every user's wrapped_dek is encrypted under the same app-wide KEK with no per-user binding. Anyone with a DB write path (SQL injection, compromised service role) can copy victim A's wrapped_dek onto attacker B's clients row; B's session then unwraps A's DEK and decrypts A's entire vault — no KEK compromise required.

Blast radius: full cross-tenant vault decryption from a single column copy.
Fix: bind AAD to client identity — wrapKey(dek, kek, utf8("dek:" + client.id)) and pass the same AAD on unwrap. A swapped blob then fails the Poly1305 tag check.
✅ Implemented 2026-06-02: wrapKey/unwrapKey now take an aad param; lib/api/dek.ts binds aad = utf8("dek:" + client.id) on wrap and unwrap, so a blob copied onto another client's row fails the Poly1305 tag check. Legacy no-AAD blobs self-heal on read (try-AAD → fallback → re-wrap with AAD → persist, tracked by new clients.dek_aad_version). Migration 20260602_dek_aad_binding.sql; tested in lib/crypto/__tests__/dek-aad.test.ts.

C-3 Critical getOrCreateUserDek write race mints two DEKs → permanent vault data loss ✓ verified ✅ Fixed

lib/api/dek.ts:58-65

Check-then-update is not atomic and the UPDATE is unconditional (no wrapped_dek IS NULL guard). Two concurrent first-use requests both read wrapped_dek = null, generate different DEKs, and the second overwrites the first. Anything encrypted under DEK#1 becomes permanently undecryptable. Fails silently.

Blast radius: every Option-A vault item/document; silent, unrecoverable data loss.
Fix: UPDATE clients SET wrapped_dek=... WHERE id=$1 AND wrapped_dek IS NULL, then re-SELECT; if another writer won, unwrap the persisted value. Or INSERT ... ON CONFLICT / advisory lock.
Status 2026-06-02: Fixed. UPDATE now guarded .is("wrapped_dek", null) (atomic compare-and-set) then re-SELECTs; a losing writer unwraps the persisted value. Data-loss race closed.

H-4 High Trustee OTP attempt counter is non-atomic → brute-force throttle is racy ✅ Fixed

app/api/trustee/unlock-verify/route.ts:38 → lib/repos/server/farewellVerificationRepo.ts:204-207

Attempts are read, compared to MAX_OTP_ATTEMPTS, then written back. Concurrent requests all read the same low count, all pass the gate, each writes count+1 (lost update). The cap on a 6-digit OTP can be exceeded in parallel — this gate releases Share C → vault MK.

Fix: UPDATE ... SET otp_email_attempts = otp_email_attempts + 1 WHERE id=$1 AND otp_email_attempts < 10 RETURNING ...; reject on 0 rows. Add an Upstash limit on unlock-verify:{requestId} and cap resends per request.
✅ Implemented 2026-06-02: incrementOtpAttempts now calls an atomic increment_otp_attempt(p_request_id, p_max) RPC (UPDATE … +1 WHERE attempts < max RETURNING); the verify route returns 429 when it yields null. A resend rate-limit (trusteeOtpResendRateLimit, 3/hr per request) caps fresh codes so the per-code cap can't be bypassed by spamming resends. MAX_OTP_ATTEMPTS = 10. Migration 20260602_otp_attempts_atomic.sql.
