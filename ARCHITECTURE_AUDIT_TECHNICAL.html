EstateVault — Backend & Data Architecture

Multi-tenant white-label estate-planning SaaS. Next.js 14 (App Router) · Supabase (Postgres + RLS + Storage + Realtime) · Stripe · Resend · Anthropic. Scope: schema, RLS, migrations, API routes, edge functions, frontend data access, E2EE.

118+ API routes ~45 migrations ~39 RLS policies 597 direct .from() calls 76/118 routes use service_role
Executive Verdict

The system is a de-facto hybrid built by accident: 64% of routes use the service_role key (RLS bypass), yet ~39 RLS policies are maintained. You pay for two authorization models and get the guarantees of neither. The data layer (tenancy, indexing, E2EE) is strong; the application/access layer lacks a spine — no DAL, no single auth boundary, no regression net. Recommendation: B (Hybrid) via a consolidated service layer inside Next.js — not NestJS, not a migration.

Section 2

Why fixing one thing breaks another

Root-cause analysis of the regression treadmill. This is a structural symptom, not bad luck.

The codebase exhibits classic shotgun surgery + missing abstraction layer. The same concerns (data access, tenant scoping, authz, validation) are reimplemented inline per route instead of factored into shared modules. A fix touches one copy; the other N copies retain the defect, so the bug "reappears" elsewhere. There is no compiler- or test-enforced single source of truth to catch the divergence.

Concrete example of divergence risk
Revenue split is computed in lib/stripe-payouts.ts calculateSplit() but invoked from both app/api/webhooks/stripe/route.ts and inline in checkout. If pricing rules change in one path and not the other, payouts disagree silently — and nothing tests the equivalence.

Change-risk matrix (expected regression frequency)

Blast radius is a function of change type. Cross-cutting concerns (schema, money, authz) have no single edit point, so they regress often — and the failure surfaces away from the edit site, defeating "test what I changed."

Empirical estimate
For changes touching shared concerns (data access, money, authz), expect a ~1-in-3 regression rate, with the defect surfacing in a non-local module. Root cause: high fan-in to duplicated logic + zero regression coverage. The "randomness" is duplication, not chance.

Trajectory
Coupling cost scales super-linearly with route count. Every new route adds another copy site and another unguarded-by-default surface. Onboarding cost rises (no canonical schema, no test net to encode intent). Untreated, velocity decays as the codebase grows.
✓ How to structurally end the treadmill
  • Introduce a DAL (repos + services). One module owns each concern; routes call it. Fix-once semantics. The repos already exist in lib/repos/* — they're just bypassed by the API.
  • Single authz boundary. Mandatory requireAuth([roles]) + lint/CI gate so no route can ship without it.
  • Regression net. Integration tests around services + a contract test asserting payout/authz invariants. CI catches divergence pre-merge.
  • Canonical schema snapshot. One generated schema.sql as source of truth; one migration naming convention.

This is precisely the work sequenced in Section 9. The cascade stops the moment shared logic has exactly one home.

Section 3

RLS Evaluation
Verdict
RLS is simultaneously overused and underused. It is not the effective security boundary today.
  • Underused in practice: 76/118 routes instantiate a service_role client → RLS bypassed. Effective boundary = per-route handwritten checks.
  • Overused conceptually: ~39 policies, several with correlated subqueries / EXISTS, maintained while mostly bypassed.
  • Perf risk (confirmed): SECURITY DEFINER helpers call unwrapped auth.uid() → re-evaluated per row.
  • Fragility: RLS is load-bearing only for ~25% browser-direct queries (doc polling, profile reads, realtime). One OR/IN bug = cross-tenant leak, no second net.

Current helper (re-runs per row):

create function public.get_partner_id() returns uuid as $$
  select id from public.partners where profile_id = auth.uid();
$$ language sql security definer stable;

Planner-cached fix (Supabase-recommended):

-- evaluate once per statement, not per row
where profile_id = (select auth.uid())
Direction
Demote RLS to defense-in-depth; make the API the authoritative boundary. Keep RLS on as a backstop, fix the auth.uid() wrapping, index policy-subquery columns.

Section 6

Scaling Risk Analysis

1× → 10×  Holds

Postgres + Supavisor pooling fine. First breakages: (1) in-memory Map-based rate limiter in lib/api/auth.ts — per-instance, useless on serverless; (2) synchronous doc-gen route → Vercel function timeout under burst.

10× → 100×  Strain

  • Connection exhaustion: per-request admin client instantiation. Enforce Supavisor/PgBouncer transaction pooling before CPU becomes the limit.
  • RLS subqueries + unwrapped auth.uid() degrade as clients/vault_items grow.
  • Doc generation (Claude, synchronous, in-request) must move to lib/queue/.

Concurrency / correctness

No idempotency keys observed on checkout/order creation → double-submit and Stripe webhook retries can create duplicate orders. Add idempotency keys + unique constraints. Realtime is per-user scoped (client-docs-${clientId}, attorney-queue-${userId}) — fine; watch multi-tab fan-out for large partners.

Section 8 — Decision

→ B) HYBRID via a service layer inside Next.js. Not NestJS. Not migration.

Keep Supabase (Postgres, Auth, Storage, Realtime, RLS-as-backstop, E2EE). Build a disciplined DAL + service layer in the existing app with one authoritative authz boundary (API). The problem is organization, not runtime.

✗ A) Keep as-is. 30+ unauthenticated routes + exec_sql = breach waiting. "Safe to scale" is false today.
✗ Add NestJS now. Textbook hybrid but premature: adds a deploy target, network hop, pooling complexity to solve a discipline problem. Justified later (persistent workers, event-driven, serverless limits).
✗ C) Migrate off Supabase. Negative ROI. Discards excellent E2EE + sound tenancy to rebuild Auth/Storage/Realtime. ~1 wasted year.
✓ B) Service layer in Next.js. Not a stack problem — a "no shared spine" problem. The Phase-2 service layer is also the seam that makes a future NestJS extraction cheap if ever needed.

Section 9

Refactor Plan — zero-downtime, ordered by risk reduction
Phase 0 — this week

Stop the bleeding

  • REVOKE all grants on exec_sql from anon/authenticated/public; ideally DROP and move DDL to migration tooling.
  • Enumerate the 30+ no-auth routes; protect or explicitly allowlist as public.
  • Replace in-memory rate limiter with the Upstash limiter everywhere (auth, checkout, documents/process).
Phase 1 — weeks 1–2

One authoritative authz boundary

  • Mandate requireAuth([...roles]); convert all 58 inline-auth routes.
  • CI/lint gate: every app/api/**/route.ts imports the guard or is on a public allowlist.
  • Single getAdminClient()/getUserClient() factory; lint-ban inline service_role instantiation.
Phase 2 — weeks 2–5

Resurrect the DAL / service layer

  • Route all DB access through lib/repos/* + new lib/services/*; each repo enforces tenant scoping in code (isolation independent of RLS state). Migrate the 597 .from() calls incrementally — API contract unchanged → zero downtime.
  • De-duplicate calculateSplit + attorney routing into single services; add invariant tests.
  • Decompose documents/process; offload generation to lib/queue/.
Phase 3 — weeks 4–6

Demote RLS to backstop, clean DB, add net

  • Keep RLS on; wrap auth.uid() → (select auth.uid()); add attorney_reviews.attorney_id index.
  • Consolidate migration naming; generate canonical schema.sql.
  • Cleanup job for plaintext quiz-answer purge window (currently manual).
  • Integration + contract tests in CI so divergence is caught pre-merge.
✓ Keep permanently
Supabase platform, column-based tenancy, Realtime, RLS-as-backstop, full E2EE stack.
Reconsider NestJS only when
Persistent workers, serverless cold-start/timeout walls queues can't fix, or background volume justifies a dedicated runtime. Phase 2's service layer makes that extraction cheap.
