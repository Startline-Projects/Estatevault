## 1. Platform Overview

EstateVault is a B2B2C white-label estate planning platform with four portals:

| Portal | Host | Audience |
|--------|------|----------|
| Consumer Site | estatevault.us | End clients |
| Partner Portal | pro.estatevault.us | Professional partners (FAs, CPAs, attorneys, insurance agents) |
| Sales Portal | sales.estatevault.us | Internal sales reps + admins |
| Attorney Portal | admin.estatevault.us | Review attorneys |

Clients create wills or trusts through a guided intake flow. Documents are generated by AI (Claude), optionally reviewed by an attorney, and delivered digitally. Clients also get an encrypted vault for storing sensitive information and farewell messages.

Partners white-label the platform under their own branding and earn revenue splits on every client purchase.

## 2. User Roles & Access

### 2.1 Role Definitions

| Role | Description | Portal Access |
|------|-------------|---------------|
| **Client** | End user who purchases estate plans and uses the vault | Consumer site + Client Dashboard |
| **Partner** | Professional who white-labels the platform for their clients | Partner Portal (Pro) |
| **Attorney** | Reviews and approves generated documents | Attorney Portal |
| **Sales Rep** | Internal team managing partner relationships and pipeline | Sales Portal |
| **Admin** | Full system access across all portals | All portals |
| **Affiliate** | Referral partner who earns commission on referred sales | Affiliate dashboard |

### 2.2 Authentication

- **Login:** Universal login page at `/auth/login` — routes to correct portal based on role.
- **Signup:** Email verification (code or link) → password setup → account creation.
- **Cross-Portal Handoff:** Encrypted session token transfers user between subdomains (e.g., `estatevault.us` → `pro.estatevault.us`) without re-login.
- **Vault PIN:** Clients set a 4-digit PIN during onboarding for vault access (separate from account password).
- **Password Recovery:** Email-based reset flow at `/auth/forgot-password`.

### 2.3 Route Guards

| Subdomain | Allowed Roles | Unauthorized Redirect |
|---|---|---|
| `localhost` (consumer) | client, public | → `pro.localhost` (partner), `sales.localhost` (sales/admin) |
| `pro.localhost` | partner | → `localhost` (client), `sales.localhost` (sales) |
| `sales.localhost` | sales_rep, admin | → `localhost` or `pro.localhost` |
| `admin.localhost` | attorney, review_attorney | → `localhost` |
| `localhost/quiz`, `/will`, `/trust` | Public (anyone) | N/A |
| `localhost/auth/*` | Public | Authenticated users → their portal subdomain |

## 4. Estate Planning Quiz

### 4.1 Quiz Flow (`/quiz`)

**View:** Animated card-based questionnaire with progress indicator.

**Questions collected (in order):**

| Step | Question | Input Type | Options |
|------|----------|------------|---------|
| A1 | State of residence | Pre-filled | Michigan (default) |
| B1 | Marital status | Choice tiles | Single / Married / Divorced / Widowed |
| B2 | Do you have children? | Yes/No | If yes → how many, any special needs? |
| C1 | Own real estate? | Yes/No | If yes → only in Michigan? |
| C2 | Own a business? | Yes/No | — |
| C3 | Estimated net worth | Choice tiles | Under $150K / $150K–$500K / $500K–$1M / Over $1M |
| D1 | Is privacy important? | Yes/No | — |
| E1 | Charitable giving plans? | Yes/No | — |
| F1 | Have existing estate plan? | Yes/No | If yes → update or start fresh? |
| G1 | Additional situations | Text | Free-form |

### 4.2 Recommendation Engine

**Logic (hardcoded):**
- **Will recommended** when ALL of: net worth under $150K, no real estate, no business ownership, privacy not important.
- **Trust recommended** for everything else.

**Output:** "Based on your answers, we recommend a [Will/Trust] package."

> **Compliance note:** Wording is always "Based on your answers..." — never "We recommend..." — platform does not give legal advice.

### 4.3 Hard Stops (Quiz Level)

| Trigger | Action |
|---------|--------|
| Special needs dependent | Halt → attorney referral |
| Multi-state real estate (trust) | Flag for attorney review |
| Business interests (trust) | Flag for attorney review |

## 8. Encrypted Vault

### 8.1 Vault Overview (`/dashboard/vault`)

**View:** PIN-protected encrypted storage for sensitive information.

**Access:** Requires 4-digit PIN (2-minute session timeout after unlock).

**Categories:**

| Category | What's Stored |
|----------|---------------|
| Estate Documents | Generated wills, trusts, POAs |
| Financial | Bank accounts, investment accounts, retirement funds |
| Insurance | Life, health, property, auto policies |
| Digital | Online accounts, passwords, subscriptions |
| Locations | Safe deposit boxes, physical document locations |
| Contacts | Attorneys, accountants, financial advisors |
| Business | Business documents, partnership agreements |
| Wishes | Final wishes, funeral preferences |

**Actions per item:**
- Add structured item (category-specific fields)
- Upload documents (encrypted at rest)
- Pin/favorite items
- Edit/delete items
- Search across items (encrypted search via blind indexing)

### 8.2 Farewell Messages (`/dashboard/vault/farewell`)

**View:** Video message recording and management.

**Purpose:** Clients can record or upload farewell video messages for loved ones, stored encrypted, to be released to trustees upon the client's passing.

**Actions:**
- Record new farewell video
- Upload pre-recorded video
- Manage existing messages (view, delete)
- Assign to specific trustees

### 8.3 Trustees (`/dashboard/vault/trustees`)

**View:** Trustee management.

**Purpose:** Clients designate trusted people who can access their vault after death.

**Rules:**
- Maximum 2 trustees per client
- Trustee name and email encrypted at rest
- Each trustee gets a scoped access level:
  - Which vault categories they can see
  - Whether they can view documents
  - Whether they can access farewell messages

**Actions:**
- Add trustee (name, email, relationship)
- Set access scope per trustee
- Remove trustee

### 8.4 Vault Subscription

**Pricing:** $99/year (auto-renewal via Stripe).

**What's included:**
- Encrypted vault storage
- Trustee access management
- Farewell message storage
- Amendment eligibility
- Annual review reminders

**Note:** Vault subscription is separate from document purchase. Clients can subscribe vault-only through a partner (basic tier) or after purchasing documents.

## 9. Trustee & Farewell Access

### 9.1 Trustee Unlock Flow

**This is the process by which a trustee gains access to a client's vault after the client's death.**

**Steps:**

1. **Trustee submits verification request** — provides identity documentation.
2. **Admin reviews request** — verifies identity (at `/attorney/farewell-verification` or `/sales/farewell-verification`).
3. **If approved → 72-hour veto window starts.**
4. **Owner notification** — client (if alive) receives urgent email with "Veto" button.
5. **If owner vetos** — access cancelled, trustee notified.
6. **If 72 hours pass without veto** — trustee access granted.
7. **Trustee receives unlock email** with time-limited link (7-day expiry).
8. **Trustee verifies identity via OTP** — 6-digit code sent to email (10-minute validity).
9. **Vault unlocked** — trustee can view scoped vault items, documents, and farewell messages.

### 9.2 Owner Veto (`/farewell/owner-veto`)

**View:** Form for client to revoke trustee access during the 72-hour window.

**Purpose:** Safety mechanism — if the client is alive and someone falsely requests access, the client can block it.

### 9.3 Trustee Vault View (`/trustee/vault`)

**View:** Read-only view of client's vault (scoped by trustee's access level).

**What trustee sees:**
- Estate documents (downloadable)
- Vault items by category (filtered by access scope)
- Farewell video playback
- Verification status

**Session:** httpOnly cookie, expires at end of access window.

## 10. Attorney Review Pipeline

### 10.1 How It Works

When a client adds the "Attorney Review" add-on ($300), their generated documents are routed to a review attorney before delivery.

### 10.2 Assignment

- If the client's partner has an in-house review attorney → assigned to that attorney.
- Otherwise → assigned from the platform's attorney pool (round-robin, filtered by state + practice area).

### 10.3 Attorney Review Dashboard (`/attorney/reviews`)

**View:** List of assigned reviews with SLA indicators.

**What's displayed:**
- Review queue (pending, in progress, completed, rejected)
- SLA deadline badges (green = on time, orange = <12 hours remaining, red = overdue)
- Filter and search
- Client and partner details per review

### 10.4 Review Detail View (`/attorney/review/[id]`)

**View:** Document review workspace.

**What's displayed:**
- Embedded document viewer
- Review form (approve/reject + notes)
- SLA countdown timer
- Client contact info
- Partner details

**Actions:**
- Approve document (stamps attorney name + bar number)
- Reject with feedback (sends back for revision)
- Upload reviewed/annotated version
- Request more information from client

### 10.5 SLA Rules

- **48-hour SLA** from assignment to completion.
- Reminder emails sent at 24 hours and 6 hours remaining.
- Overdue reviews flagged in dashboard.

### 10.6 Attorney Revenue

- Attorney review fee: $300 (100% goes to attorney — platform takes $0).
- Fee range configurable per attorney: $150–$1,500.
- Tracked in `/attorney/commission` view.

## 17. Document Generation

### 17.1 AI Engine

- **Model:** Claude Sonnet 4.5 (Anthropic)
- **Process:** Intake answers → template prompt → Claude generates personalized legal language → DOCX output
- **Templates:** Michigan-specific (will, trust, pour-over will, POA, healthcare directive)

### 17.2 Generation Pipeline

1. Client completes payment → Stripe webhook fires
2. Order status → "paid"
3. Documents queued for generation
4. Background processor picks up job
5. For each document type in the package:
   - Load Michigan-specific template
   - Build prompt from intake answers
   - Call Claude API
   - Generate DOCX file
   - Upload to Supabase Storage (encrypted)
6. If attorney review requested → status = "review" → route to attorney
7. If no review → status = "delivered" → email client download link

### 17.3 Document Versioning

- `template_version` tracks which template was used (e.g., "2025-01")
- `version` integer increments per amendment
- `parent_document_id` links amended documents to originals
- `superseded_by` marks old versions when new ones are generated

### 17.4 Document Sealing

Once client confirms they've executed (signed) documents, they're marked as sealed. Sealed documents cannot be regenerated — only superseded via amendment.

## 20. Security & Encryption

### 20.1 Vault Encryption Architecture

**End-to-end encryption** — Supabase (server) never sees plaintext vault data.

**Key hierarchy:**
1. Client passphrase → Argon2 KDF → 32-byte master key
2. Master key split via **Shamir Secret Sharing (3-of-3)**:
   - Share A: Stored on client device
   - Share B: Server-wrapped, encrypted at rest
   - Share C: Client-managed recovery
3. **Data Encryption Key (DEK):** 256-bit per vault, wrapped with master key
4. **Per-item encryption:** DEK + AEAD (ChaCha20-Poly1305)

### 20.2 Encrypted Search

Blind indexing on item labels — allows search without decrypting all items.

### 20.3 Trustee Key Release

When trustee access is granted, the system reconstructs the master key from shares to decrypt vault content for the trustee session.

### 20.4 Recovery

Mnemonic phrase (BIP39) recovery flow at `/recover` — client enters recovery phrase to derive new passphrase.

### 20.5 Other Security Features

- Row-Level Security (RLS) in Supabase — users can only access their own data
- Vault PIN (separate from password) with rate limiting
- 2-minute vault session timeout
- httpOnly cookies for authentication
- CSP reporting endpoint
- Clipboard scrubbing for sensitive data
- XSS sanitization
