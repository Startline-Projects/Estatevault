The Core Finding

This is not a design problem — it's an adoption problem. The right abstractions already exist and are well built: createAdminClient (lib/api/auth.ts), requireAuth, lib/email.ts, authedFetch, and Zod schemas. The routes just don't use them — they copy-paste instead. Folders look like clean layers; the actual dependency flow is ad-hoc. Most fixes below are mechanical, not redesign.

4 · Auth / role guard repeated inline

20 files High

20 routes inline the same dance: getUser() → fetch profiles.user_type → 401/403. Shared requireAuth() exists; only ~7 routes use it. Error strings are inconsistent: "Unauthorized" ×63, "Not authenticated" ×11, "unauthorized" ×4.

Why it blocks scale: change one auth rule — add a role, an audit log, a lockout — and you edit 20 places. Inconsistent error strings break client-side error handling.

5 · No shared data-access layer

61 files · 122 calls High

61 UI files fire raw inline fetch("/api/...") — 122 calls total. The shared wrapper lib/api/authedFetch.ts (handles 401 → token refresh → retry) is used in exactly 1 file: app/attorney/review/[id]/page.tsx.

Why it blocks scale: 60 screens have no 401-refresh = random logouts on expired tokens. No shared loading / error / retry behavior. A URL or header change means hunting 122 call sites.

6 · Pricing magic numbers inline

6 files Medium

Fixed prices (CLAUDE.md: "never change") are hardcoded as raw cents across checkout routes instead of one constants file.

7 · App-URL env var has 3 names

12 files Medium

Same concept, three names: NEXT_PUBLIC_SITE_URL (10 files) vs NEXT_PUBLIC_APP_URL (1) vs NEXT_PUBLIC_BASE_URL (1).

Why it blocks scale: this is a correctness bug, not just DRY — password-reset and verify links can be built from the wrong domain depending on which route runs.

8 · No shared types / contract

all layers Medium

No generated Supabase DB types (database.types.ts absent). Zod schemas in lib/validation/schemas.ts are imported by 1 route, 0 components. Request/response shapes are redefined on each side of every fetch.

Why it blocks scale: rename a DB column and nothing fails at compile time — it breaks silently at runtime. Frontend and backend payloads drift with no enforcement.

Verdict

The layers exist as folders — app/ pages, app/api/ routes, lib/ domain logic — and the genuine domain logic (document generation, templates, revenue-split math, validation) is properly extracted and kept out of React components. That layer is solid.

But the UI ↔ backend boundary is not enforced. The real flow is UI → (raw fetch | direct DB), not UI → data layer → API → lib. 67 routes copy the admin client, 51 screens query the DB directly, and the one token-refresh wrapper is used once.

The good news: nearly every fix is adoption, not redesign. Items 1, 2, 4, 5 are mechanical and low-risk — do them first for a fast structural win. Item 6 (UI direct-DB) is the real architecture work and the main security exposure — phase it deliberately.
