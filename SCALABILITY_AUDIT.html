Implemented vs Remaining — in plain English

✅ Done — you don't need to touch these again

  • B1 · One database admin client (was 67 copies). Every route used to paste its own copy of the function that opens a privileged DB connection. They now all import one shared createAdminClient from lib/api/auth.ts. Why it matters: change the DB connection once and all 120 routes follow — no more hunting 67 files.
  • B6 · One price list (was scattered). Prices ($400 will, $600 trust, etc.) were typed as raw numbers in many checkout files. Now there's a single lib/orders/pricing.ts (PRICES) everything reads from. Why it matters: no risk of charging a wrong amount because one file was missed.
  • B8 · The code now knows the database shape. Generated types (types/db.generated.ts) plus Zod validation on ~59 routes. Why it matters: rename a DB column and the build breaks loudly at compile time instead of failing silently for a user.
  • Serverless state fixed (M-6). Email-verification codes used to live in one server's memory; on Vercel a user's "verify" click could land on a different server that never saw the code, so it randomly failed. Now codes live in shared Redis — every server sees them. Why it matters: verification works reliably as traffic spreads across servers.
  • Safe under retries & double-clicks. Stripe transfers and affiliate payouts now carry idempotency keys, and the trustee security-code counter is a single atomic database step. Why it matters: retries and parallel requests can't double-pay or blow past limits when load is high.
  • B7 · One name for the app's URL — RESOLVED (2026-06-02). The three env-var names were replaced by a single getAppUrl() helper (lib/config/appUrl.ts) used in all 13 places. Why it matters: password-reset / verify links can no longer be built from the wrong domain.

The Core Finding

This is not a design problem — it's an adoption problem. The right abstractions already exist and are well built: createAdminClient (lib/api/auth.ts), requireAuth, lib/email.ts, authedFetch, and Zod schemas. The routes just don't use them — they copy-paste instead. Folders look like clean layers; the actual dependency flow is ad-hoc. Most fixes below are mechanical, not redesign.

5 · No shared data-access layer

~50 screens on api-client · 0 raw internal fetch High ✅ Resolved — lib/api-client/

61 UI files fired raw fetch("/api/..."); the authedFetch wrapper was used in 1 file. Resolved 2026-06-02 (verified): a typed lib/api-client/ layer (11 domain modules over client.ts → authedFetch) now backs the screens — ~50 screens import it and there are 0 raw internal /api/ fetches left in client code. Why it blocked scale: screens had no 401-refresh = random logouts. Now: one place to change a URL/header, shared error handling, and token-refresh everywhere.

Verdict

The layers exist as folders — app/ pages, app/api/ routes, lib/ domain logic — and the genuine domain logic (document generation, templates, revenue-split math, validation) is properly extracted and kept out of React components. That layer is solid.

But the UI ↔ backend boundary is not enforced. The real flow is UI → (raw fetch | direct DB), not UI → data layer → API → lib. 67 routes copy the admin client, 51 screens query the DB directly, and the one token-refresh wrapper is used once.

The good news: nearly every fix is adoption, not redesign. Update 2026-06-02: items 1, 4, 8 are now done and 2, 3 mostly done — the admin-client copies are gone (67→0), pricing is centralised, DB types + Zod boundaries are in place, and serverless state (email verification) moved to Redis with idempotent payments/atomic OTP for concurrency safety. Update 2026-06-03 (commit 9a0d97a): item 6 (B2) is now done — every browser-side screen runs through an API route; the UI ↔ backend boundary is enforced (UI → api-client → API → repo → DB). Only 2 server-side RLS auth gates remain by design. The architecture/security track is effectively complete; lower-value B3 per-send email branding is the only optional item left.
