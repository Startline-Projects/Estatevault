The intended layering — UI → api-client → API route (auth + Zod) → repo → DB — is real and followed by the large majority of the app. That single-path discipline is what makes a codebase scale: one place to change DB access, one place for auth, one place for validation.

2 · What's genuinely strong

  • Type safety is excellent. Across all of lib/ + app/: 7 any/as any total, 0 @ts-ignore/@ts-expect-error. TypeScript strict is doing its job — runtime surprises have few places to hide.
  • Thin routes, fat repos is the norm. A typical route reads top-to-bottom in ~15 lines: requireAuth → safeParse → call a repo → ok(). DB chains live in lib/repos/server/*, not inline.
  • Auth is layered and mostly intentional, not ad-hoc: requireAuth (77 routes, cookie + mobile Bearer), requireClientUser (13, vault/crypto), requireTrusteeSession (trustee OTP identity), CRON_SECRET (6 cron), Stripe webhook signature. Most "no-requireAuth" routes are guarded by one of these or are legitimately public (pre-account checkout, signup, branding).
  • Money stays integer cents, fixed prices/splits honored, hard stops present. Business law intact.
  • Validation at the boundary — 76 central Zod schemas; a unit test forbids inline z.object() in routes.
  • Real test coverage on the dangerous parts (crypto, splits, idempotency) plus 24 e2e specs including B2 endpoint guards.

3.3 · Auth drift on marketing routes fixed ✅ Fixed · e2e4938

The 4 marketing asset routes (materials, flyer, one-pager, script-card) hand-rolled createClient().auth.getUser() + a manual 401. They now use requireAuth(undefined, req) — same "any logged-in user" semantics, plus mobile Bearer support, a consistent 401, and they follow future requireAuth changes.

The rest of the ~10 raw-getUser routes were deliberately left (not drift): checkout/* (guest-capable purchase — requireAuth would block pre-account checkout), documents/download-by-session (Stripe-session-token auth, not a user), auth/welcome (needs raw user.user_metadata), and share (E2EE-vault feature; requireAuth vs requireClientUser is a separate small decision).

quizSessionRepo took row: Record<string, unknown>, which the typed Supabase client rejected — the one error keeping npx tsc --noEmit red. 3.5 · webhooks/stripe split into a thin router + handlers ✅ Fixed · 98368ae

The webhook was an 805-line file that moved money and provisioned accounts (one handler ~320 lines). The per-product handlers were extracted verbatim into lib/webhooks/stripe/* (resolveOrCreateGuestClient, handleAttorneyReview, handleAmendmentCheckout, handleVaultSubscriptionCheckout, handleDocumentCheckout). Behaviour preserved by construction — byte-identical moves, proven by tsc + the full suite (no logic changed).

The route is now a 156-line router: verify signature → idempotency guard → dispatch by event.type → call the lib handler. Each heavy handler is now independently importable + testable. The phase0-security guard was extended so the no-listUsers() check and the find_auth_user_by_email RPC assertion follow the code into the new handlers.

Recommended follow-up (not blocking): add event-simulation characterization tests per handler (mock a checkout.session.completed → assert the order/split/documents/payout writes). Cheap now that the handlers are isolated; was impractical as one 805-line function.

3.6 · custom_review_fee reviewed — NOT a pricing violation ✅ Decided · keep as-is

Flagged because a partner-editable review fee looked like it broke the "$300 attorney review, partners can't change it" rule. Tracing lib/attorney-review/routing.ts shows it doesn't: every review is charged the fixed DEFAULT_REVIEW_FEE_CENTS ($300) except when the partner is a law firm reviewing with its own in-house attorney — there the fee is custom_review_fee || $300 and that firm keeps 100%. That's the firm pricing its own legal service, not EstateVault's $300 product. The fixed-pricing rule holds.

Decision (product owner, 2026-06-03): leave as-is — the $300 rule is correct and intact; custom_review_fee is the legitimately separate in-house-attorney fee. The only residual is a minor least-privilege nicety (any partner can write the field, but routing ignores it for non-attorney partners, so no pricing effect). Accepted as harmless.

4 · Is it scalable?

Architecturally, yes. The properties that let a codebase grow without getting more fragile are present:

  • A single data-access layer. 21 repos; the DB schema can change in one place. ✅
  • An enforced input boundary. 76 Zod schemas; client input is parsed, not trusted. ✅
  • Centralized auth with a clear (if multi-path) set of guards. ✅
  • Strong types so refactors are caught at compile time, not in production. ✅
  • A mobile frontend is now realistic — requireAuth already does Bearer tokens and the api-client layer is the contract a mobile app would reuse. With B2 complete, every screen's data path is a reusable endpoint.

What would limit scale if left unattended:

  • Drift seeds (a template someone copies next time) erode the "one way to do it" property. The two biggest pools — the half-migrated screens (3.1) and the inline admin clients (3.2) — have both since been converged.
  • 148 routes is a lot of surface; thin-route/fat-repo discipline is what keeps it cheap. It's holding — keep enforcing it (the no-inline-z.object and no-as any tests help).
  • The Stripe webhook used to be the one place where "scale" meant "complexity that's hard to test" — now decomposed into a thin router + isolated handlers (3.5), so each piece is testable on its own.
Net: a mid-stage codebase that has done the hard structural work (layering, types, validation, repos) and is now in the finish-and-converge phase, not the redesign phase. That's a healthy place to be.
