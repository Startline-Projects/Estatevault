1.1 · C-2 Open Bind the DEK wrap to client identity (AAD) Effort L

lib/crypto/keyManager.ts · lib/crypto/aead.ts · lib/api/dek.ts

Problem: a user's wrapped vault key has no "owner" stamped on it; copying it onto another account row lets that account decrypt the victim's vault.

  1. Change wrapKey(mk, kek) → wrapKey(mk, kek, aad) and unwrapKey(env, kek) → unwrapKey(env, kek, aad); pass aad through to encryptBytes/decryptBytes.
  2. In dek.ts build const aad = utf8("dek:" + client.id) and pass it at every wrap (~:51) and unwrap (~:55, ~:80).
  3. Migration path: on unwrap, try with AAD; if the tag fails, retry once with no AAD, then re-wrap with AAD and persist. Gate behind a dek_aad_version column so it runs once per user. (Or a one-off backfill if downtime is OK.)

Phase 3 — Make racy gates atomic

3.1 · H-4 Partial Atomic OTP attempt counter Effort M

app/api/trustee/unlock-verify/route.ts (MAX_OTP_ATTEMPTS, now 10) · lib/repos/server/farewellVerificationRepo.ts

Problem: read-then-write counter lets the per-code guess cap be exceeded by parallel requests. Cap is per code (resets to 0 on each new/burned code, not a lifetime lock) and each code already expires. MAX_OTP_ATTEMPTS was raised 5 → 10 for typo tolerance — security-neutral, each new code is still 1-in-1,000,000.

  1. Replace read-then-write with an atomic conditional update — a Postgres RPC or .update({...}).lt("otp_email_attempts", MAX).select() that increments only while under the cap and returns the affected row; reject on 0 rows.
  2. Defense in depth: add an Upstash limit keyed on unlock-verify:{requestId}.
  3. Resend rate-limit (required): issuing a new code resets the counter, so spammed resends = unlimited fresh 10-guess batches. Cap resends per request (e.g. 3/hour) in the OTP-issue route — the per-code cap is meaningless without this.
⚠️ Steps 1 and 3 are both required — not either/or. Atomic counter alone → one code caps at 10, but resend gives unlimited fresh batches. Resend cap alone → resends limited, but the race lets one code yield 50+ guesses in a millisecond. Fix one, the attacker uses the other. With both: ~10 guesses × ~3 resends/hour ≈ 30/hour against a 1,000,000 space = safe. Ship together.
Test: fire N parallel verifies — attempts never exceed MAX_OTP_ATTEMPTS; resend test asserts the resend limit holds.
