1. Platform Overview

EstateVault is a B2B2C white-label estate planning platform with four portals:

PortalHostAudience
Consumer Siteestatevault.usEnd clients
Partner Portalpro.estatevault.usProfessional partners (FAs, CPAs, attorneys, insurance agents)
Sales Portalsales.estatevault.usInternal sales reps + admins
Attorney Portaladmin.estatevault.usReview attorneys

Clients create wills or trusts through a guided intake flow. Documents are generated by AI (Claude), optionally reviewed by an attorney, and delivered digitally. Clients also get an encrypted vault for storing sensitive information and farewell messages.

Partners white-label the platform under their own branding and earn revenue splits on every client purchase.

2.2 Authentication

  • Login: Universal login page at /auth/login — routes to correct portal based on role.
  • Signup: Email verification (code or link) → password setup → account creation.
  • Cross-Portal Handoff: Encrypted session token transfers user between subdomains without re-login.
  • Vault PIN: Clients set a 4-digit PIN during onboarding for vault access (separate from account password).
  • Password Recovery: Email-based reset flow at /auth/forgot-password.

9. Trustee & Farewell Access

9.1 Trustee Unlock Flow

Process by which a trustee gains access to a client's vault after passing:

  1. Trustee submits verification request — provides identity documentation.
  2. Admin reviews request — verifies identity.
  3. If approved → 72-hour veto window starts.
  4. Owner notification — client (if alive) receives urgent email with "Veto" button.
  5. If owner vetos — access cancelled, trustee notified.
  6. If 72 hours pass without veto — trustee access granted.
  7. Trustee receives unlock email with time-limited link (7-day expiry).
  8. Trustee verifies identity via OTP — 6-digit code sent to email (10-minute validity).
  9. Vault unlocked — trustee can view scoped vault items, documents, and farewell messages.

9.2 Owner Veto (/farewell/owner-veto)

Purpose: Safety mechanism — if the client is alive and someone falsely requests access, the client can block it.

9.3 Trustee Vault View (/trustee/vault)

What trustee sees:

  • Estate documents (downloadable)
  • Vault items by category (filtered by access scope)
  • Farewell video playback
  • Verification status

Session: httpOnly cookie, expires at end of access window.

17. Document Generation

17.1 AI Engine

  • Model: Claude Sonnet 4.5 (Anthropic)
  • Process: Intake answers → template prompt → Claude generates personalized legal language → DOCX output
  • Templates: Michigan-specific (will, trust, pour-over will, POA, healthcare directive)

17.2 Generation Pipeline

  1. Client completes payment → Stripe webhook fires
  2. Order status → "paid"
  3. Documents queued for generation
  4. Background processor picks up job
  5. For each document type in the package:
    • Load Michigan-specific template
    • Build prompt from intake answers
    • Call Claude API
    • Generate DOCX file
    • Upload to Supabase Storage (encrypted)
  6. If attorney review requested → status = "review" → route to attorney
  7. If no review → status = "delivered" → email client download link

17.3 Document Versioning

  • template_version tracks which template was used (e.g., "2025-01")
  • version integer increments per amendment
  • parent_document_id links amended documents to originals
  • superseded_by marks old versions when new ones are generated

17.4 Document Sealing

Once client confirms they've executed (signed) documents, they're marked as sealed. Sealed documents cannot be regenerated — only superseded via amendment.

20. Security & Encryption

20.1 Vault Encryption Architecture

End-to-end encryption — Supabase (server) never sees plaintext vault data.

Key hierarchy:

  1. Client passphrase → Argon2 KDF → 32-byte master key
  2. Master key split via Shamir Secret Sharing (3-of-3):
    • Share A: Stored on client device
    • Share B: Server-wrapped, encrypted at rest
    • Share C: Client-managed recovery
  3. Data Encryption Key (DEK): 256-bit per vault, wrapped with master key
  4. Per-item encryption: DEK + AEAD (ChaCha20-Poly1305)

20.2 Encrypted Search

Blind indexing on item labels — allows search without decrypting all items.

20.3 Trustee Key Release

When trustee access is granted, system reconstructs master key from shares to decrypt vault content for trustee session.

20.4 Recovery

Mnemonic phrase (BIP39) recovery flow at /recover — client enters recovery phrase to derive new passphrase.

20.5 Other Security Features

  • Row-Level Security (RLS) in Supabase — users can only access their own data
  • Vault PIN (separate from password) with rate limiting
  • 2-minute vault session timeout
  • httpOnly cookies for authentication
  • CSP reporting endpoint
  • Clipboard scrubbing for sensitive data
  • XSS sanitization
