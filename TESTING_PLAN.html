1 · What we're testing & why

B2B2C estate platform, five roles (client · partner · sales rep · review attorney · admin), real money (Stripe checkout + Connect payouts + revenue splits), E2EE vault crypto, and hard business rules. Recent work: all UI off direct DB (B2), one admin client (3.2), tightened auth (3.3) + types (3.4), decomposed the Stripe webhook (3.5).

Goals, in priority order:

  • Money is never wrong — correct prices, correct splits, no double-charge, payouts to the right account.
  • Access control holds — each role sees only what it should; the new B2 endpoints reject anonymous + cross-tenant callers.
  • Hard rules never break — fixed pricing, hard stops (special-needs / irrevocable trust → attorney referral), acknowledgment before generation, "never say death."
  • Core journeys work end-to-end — quiz → checkout → generation → vault; partner onboarding; attorney review; trustee unlock.
  • No regressions from the refactors — converted screens still load + act through their new endpoints.

2 · Test harness

  • Playwright — playwright.config.ts, 203 e2e in 24 specs. Multi-portal projects on *.localhost:3100: app (consumer), pro, sales, admin. Tag a test @pro/@sales/@admin to route it; untagged → consumer.
  • Dev server auto-starts (next dev --port 3100) from .env.test. Auth helper loginAs() + adminClient(); fixtures one user per role.
  • Vitest unit suite — 378 tests (split math, crypto, idempotency, Zod boundaries, phase0 security guards).
⚠ Environment caveat: .env.test and .env.local point at the same Supabase project — a shared DB. Reset/seed scripts hard-guard to only touch @estatevault.test users and require TEST_ALLOW_SHARED_DB=1. Non-destructive tests run with no seeding; authenticated tests need seeding first (deliberate).

4 · Areas & cases — P0 money/security/rules · P1 journeys · P2 secondary

A · Access control & auth boundary P0

  • Every B2 GET endpoint rejects anonymous (covered: b2-partner-api, incl. client/quiz-latest, auth/login-routing, sales/my-platform-commission).
  • New mutating endpoints reject anonymous: partner/me PATCH, partner/{invite-client,apply-promo,logo,certify}, profile/me PATCH, sales/leads/[id], sales/attorney-verification, sales/partners/[id] (+/apply-promo).
  • Cross-tenant: partner can't read another partner's client (403/404); sales rep can't read a partner they didn't create (404).
  • Role-host lockout: portal accounts blocked on the consumer host + vice-versa (re-verify after the auth/login → login-routing conversion).
  • Mobile Bearer works on the marketing routes now on requireAuth (3.3).

B · Consumer journey: quiz → checkout → documents P0P1

  • Hard stops (special-needs / irrevocable trust) halt generation → attorney referral; assert no order/document row is created.
  • Fixed pricing at checkout: Will $400 · Trust $600 · Attorney Review +$300 · Amendment $50 — assert the Stripe session amount, not the UI label.
  • Acknowledgment required before any document generates.
  • Quiz prefill via GET /api/client/quiz-latest; success pages poll check-status; correct doc set per product.
  • "Based on your answers" / "never say death" copy.

C · Payments, splits, payouts (Stripe) P0

  • Split math: standard $300/$400, enterprise $350/$450 (unit: calculate-split, 17 cases).
  • Webhook idempotency — same event.id twice → duplicate:true no-op.
  • Webhook routing after 3.5 — dispatch by metadata to the right lib/webhooks/stripe/* handler; amendment never generates a doc set (H-1).
  • Partner payout: connected → transfer sent; unconnected → pending. Affiliate mirrors when no partner.
  • Attorney fee routing — EstateVault counsel fixed $300; partner in-house → custom_review_fee || $300 (assert non-attorney partner's value is ignored — the 3.6 invariant).
  • Vault renewal / past_due+dunning / cancelled.

D · Partner portal P1

  • Onboarding steps 1–7 + vault variants (B2): plan + platform-fee checkout, branding + logo upload, pricing, domain, email, payouts, finish + certify + invite.
  • Promo comp is server-validated; a forged partner/me PATCH cannot set financial flags (whitelist excludes them).
  • Settings save via partner/me + profile/me + partner/logo. Revenue math; white-label host lockout.

E · Sales & attorney portals P1

  • Sales dashboard + partner-detail twins via the new sales/* endpoints; numbers in dollars (post-3.1 fix).
  • Lead "mark contacted" → sales/leads/[id]; attorney activate/reject → sales/attorney-verification.
  • Commission: rep platform-fee model vs admin view. Attorney queue: approve/changes; cannot review own client.

F · Client vault & E2EE P0P1

  • Crypto: bootstrap, bundle, passphrase + recovery rotation; PIN separate from password.
  • Vault CRUD scoped to the owning client. Trustee unlock — OTP (cap 10), 72h window, identity verification; no cross-client access.
  • Funding checklist + client settings via client/funding-checklist + client/settings (B2).

G · Public / misc P2

  • Marketing PDFs render for a logged-in partner, 401 anonymous (now requireAuth, 3.3). Affiliate link click tally. Public-page smoke (no 5xx).

6 · Execution strategy

  • Stage 0 — unit gate (no DB): npm test → 378 pass. Run first/always.
  • Stage 1 — non-destructive e2e (no seeding): anonymous guards + public smoke. Proves the harness + boundary.
  • Stage 2 — authenticated e2e: npm run test:db:seed:shared then npm run test:e2e (with the harness fixes above).
  • Stage 3 — money-path focus (Stripe test mode): stripe-webhooks, partner-revenue, vault-subscription, attorney-review; verify amounts/splits/payouts against calculateSplit.
Pass/fail bar: all P0 green is release-blocking; P1 green for the journeys a release touches; P2 may carry known-skips with a note.
